# 릴레이 신뢰 경계

**요약: 기본 설정에서 제3자가 당신의 카카오톡 대화 평문을 볼 수 있습니다.**

## 왜 릴레이가 필요한가

카카오톡 채널의 오픈빌더 스킬 서버는 **공개 HTTPS 엔드포인트**를 요구합니다.
Hermes 게이트웨이는 보통 개인 머신이나 홈서버에서 돌기 때문에 카카오가 직접
호출할 주소가 없습니다. 릴레이가 그 사이에 서서:

1. 카카오의 스킬 콜백을 공개 주소로 받고 (~5초 내 ACK),
2. 게이트웨이가 SSE로 구독하는 스트림에 메시지를 넣고,
3. 게이트웨이가 나중에 보낸 응답을 카카오로 전달합니다.

## 무엇이 노출되는가

```
카카오톡 사용자 ──▶ 카카오 ──▶ [릴레이] ──SSE──▶ 당신의 Hermes 게이트웨이
                                  ▲                        │
                                  └──────── 응답 ───────────┘
```

릴레이를 통과하는 데이터:

| 데이터 | 노출 |
|---|---|
| 사용자가 보낸 메시지 본문 | 평문 |
| 에이전트가 생성한 응답 본문 | 평문 |
| 카카오 `botUserKey` (사용자 식별자) | 평문 |
| 채널 ID | 평문 |
| 릴레이 세션/페어링 토큰 | 릴레이가 발급·보관 |

TLS는 전송 구간만 보호합니다. 릴레이 **서버 자체**는 모든 내용을 복호화된 상태로
처리합니다. 종단간 암호화는 없습니다.

## 기본값

`KAKAO_RELAY_URL`의 기본값은 `https://k.tess.dev/` 입니다. 이것은
`openclaw-kakao-talkchannel-plugin`이 쓰던 것과 동일한 **제3자 운영 릴레이**이며,
이 저장소의 관리자가 운영하지 않습니다. 운영 주체·보존 정책·로그 정책은
**확인되지 않았습니다.**

## 판단 기준

기본 릴레이를 그대로 써도 되는 경우:

- 개인 실험, 데모, 프로토타입
- 대화 내용이 유출돼도 실질적 피해가 없는 경우

**쓰면 안 되는 경우:**

- 고객·환자·직원 등 제3자의 개인정보가 오가는 경우
- 사내 기밀, 미공개 정보
- 개인정보보호법상 처리위탁 고지·동의가 필요한 서비스
- 상용 서비스 전반

## 자체 릴레이로 옮기기

```bash
export KAKAO_RELAY_URL="https://relay.mycompany.example/"
```

또는 `~/.hermes/config.yaml`:

```yaml
gateway:
  platforms:
    kakaotalk:
      enabled: true
      extra:
        relay_url: https://relay.mycompany.example/
```

릴레이가 구현해야 하는 와이어 프로토콜은
[relay-wire-protocol.md](relay-wire-protocol.md)에 정리되어 있습니다. OpenClaw
저장소의 `docs/RELAY_SELF_HOSTING.md`와 `docs/CUSTOM_DEPLOY_GUIDE.md`도 참고
자료입니다.

> 자체 호스팅은 신뢰 경계를 릴레이 운영자에서 **당신**으로 옮길 뿐, 없애지는
> 않습니다. 카카오 자체는 여전히 모든 메시지를 봅니다 — 이것은 카카오톡 채널을
> 쓰는 이상 피할 수 없습니다.
